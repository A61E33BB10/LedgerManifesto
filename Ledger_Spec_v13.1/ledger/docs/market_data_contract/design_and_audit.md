### NAZAROV
DESIGN:
BOUNDARY HELD: the surface where market and reference data cross into the closed system — everything upstream of `ingest` (tex 2606-2642) plus the one artifact flowing the other way. The contract COMPLETES §8 at that surface; no internal semantics move.

TWO DESIGNS FOR STAMPING AUTHORITY + PROJECTION INTERFACE, and the pick.
Design A — THE DOOR STAMPS. `ingest` (which already receives `Ledger`) authors the stamp at admission: stamp(u, source, t_obs) = (logged per-(unit,source) dissemination convention) applied to (the effective-order chain as-of t_obs, from the committed log). The read-only basis projection is the door's own read path, additionally EXPORTED so any external ingestion layer can pre-stage, correlate vendor CA references, and monitor — but the door's stamp is the one that binds. External layers assert exactly one thing: their source's dissemination convention (that is TA-BASIS, unchanged in scope).
Design B — GATEWAYS STAMP AGAINST A CITED PROJECTION VERSION. The ledger publishes the projection as a signed, versioned, content-addressed feed; per-vendor gateways stamp; each StampedObs cites the view id it stamped against; `ingest` verifies stamp-consistency-with-cited-view and bounds view staleness, refusing otherwise.
PICK: A. Reasons: (1) zero-trust — under B the stamp is an external assertion the door must re-verify anyway, so B's distribution either widens TA-BASIS to "the gateway computed correctly against a stale-bounded view" (a hole) or duplicates the computation (waste); A keeps stamp authorship inside the trust boundary and shrinks the provider assertion to the minimal fact the provider actually knows. (2) B has a standing race — the window between served view and commit — that A closes by construction: the door stamps against the committed log at admission. (3) Minimalism: A adds no new artifact classes; B adds view-citation, staleness bounds, and a verification protocol. A retains B's one virtue as a SHOULD: a gateway MAY pre-stamp citing a view id; the door recomputes; divergence is logged as a TA-BASIS detection signal (it is precisely the signature of a boundary committed in between).

G1 — NAMED PRINCIPLE (working name "Stamping authority"; plain fallback "single stamper"): the ledger's boundary chain is the single source of basis truth; every stamp is derived from that chain via a read-only projection plus a logged (unit, source) convention; no component outside the ledger authors, defaults, or adjusts a basis coordinate. THE PROJECTION AS AN INTERFACE: operations chainOf(u) → [(bid, t_eff, prec)] in effective order; tipOf(u) → BasisId (= usBasis, equal to last of chainOf by the tip weld); asOf(t) selecting the knowledge-time view; correlate(notice content-address) → Maybe BasisId so a gateway maps a vendor's CA reference to a boundary id without seeing operator content. EXPOSES NO OpSpec, no parameters, no arrows — arithmetic-free by construction, which makes "no CA logic outside the Ledger" a consequence of information hiding, not a policy. Consistency semantics: a projection of the log like every other view (derived, discardable, never authoritative, rebuilt by replay — the existing cache discipline, P8); every answer cites its view id = log-prefix content address; a served view is immutable; retro-effective boundaries appear only in later views (monotone per knowledge axis). Read-only structurally: no write path exists. Consumers: any ingestion layer (in-process, gateway, batch loader) and reconciliation/diagnostics.

G2 — PROVIDER/INGESTION OBLIGATIONS (numbered): O1 a raw observation MUST carry value (exact), t_obs, source id, and a unit reference resolvable to a registered UnitId — nothing more is demanded of providers. O2 providers are NOT required to be CA-aware: a pre-adjusting provider and a lagging one are both representable because the stamp says WHICH basis the value is in — mQuoteEx generalised from Boolean to the coordinate. O3 each (unit, source) pair MUST have a dissemination convention on file, entered as a logged observation event under the existing canonical-handler machinery (versioned like basis notices, no new state home); owner: market-data operations, under TA-BASIS, reconciled daily against vendor-published adjustment factors. O4 the door attaches the stamp per G1 and records in the attested envelope the view id and convention version used — every stamp replays deterministically as f(view id, convention version). O5 a datum whose basis cannot be determined (no convention, ambiguous convention across a pending-transition) is stamped Pending-equivalent and quarantined (W2/W3), never given a guessed basis; an unresolvable unit reference is refused outright (existing rule, tex 2626). O6 attestation envelope as already specified (provider key, signature, content address); unsigned feeds enter only through a gateway that counter-signs under a named key with its own scoped trust assumption. O7 vendor value corrections are new observations superseding by (source, unit, t_obs); they never mutate; basis re-coordination is exclusively the ledger's re-stamp event (§8.6).

G3 — DIRECTION, one sentence: the ledger publishes exactly one thing to the market-data layer — the read-only basis projection, a view of the log — and accepts exactly one thing from it — raw observations that the door itself stamps and seals; the market-data layer never writes ledger state, and the ledger never trusts a provider-asserted basis beyond the admission checks it performs itself.

G5 — OPERATIONAL MODES, one contract: real-time = stamp as-of t_obs (≈ current view); end-of-day batch = the identical call per row with asOf(each row's t_obs), so backfill and live share one code path and batch is not a second ingestion semantics; multi-vendor = one convention per (unit, source); disagreement on VALUE within one basis partition is data quality, out of scope; disagreement on BASIS is different stamps = W3 partitions, in scope and resolved by the stamp. WORKED EXAMPLE (two consumers, opposite sides): vendor publishes 100 at t1 (pre-split) and 50.5 at t2 (post-split, source publishes adjusted). Consumer A opens withSnapshot before τ commits: β(u)=b3, consumes 100 @ b3 — coherent. Consumer B opens after: β(u)=b4; the t1 observation, stamped b3, transports along the unique arrow to 50; the t2 observation is stamped b4 by its convention. Neither consumer can pair across; the naive "latest value 100" is a snapPrice refusal, and the phantom never forms — the seam catches it. STATED CONSEQUENCE: arrows are declared once, in boundary transactions, applied by the one evaluator; the projection is arithmetic-free; therefore no external component even POSSESSES the information needed to adjust — CA logic cannot be duplicated outside the Ledger by a well-meaning gateway, let alone a malicious one.

G4 — GENERALISATION TABLE (raw datum / stamp coverage / operator classes / fail-closed): (1) Spot & quotes: singleton stamp {u↦b}; Scale, Shift, Subst, Terminal; undeclared kind at a boundary → no arrow → W2/W2'. (2) Implied vol AS OBSERVATIONS: value plus labelled coordinates (absolute strike, expiry, moneyness/delta); each coordinate's basis-sensitivity is DECLARED DATA in the datum-kind declaration — absolute-strike coordinates carry the boundary's price action under Scale, moneyness declared invariant, THE VOL VALUE IS NEVER TRANSFORMED and no parametrization is mandated (strictly no modelling; whether the surface is sticky-anything is the black box's interior); a coordinate with undeclared sensitivity blocks the kind fail-closed. (3) Index levels: derived observations with joint stamp over {index}∪constituents; never transported (re-derivation canon); pointwise mismatch → Unpriced, never a silent level (the divisor example, tex 2950-2964, becomes an instance of the contract). (4) Compositions/weights/divisors: basis notices under W4 declaring Recompose on the composition kind; joint stamp; composition change without a maintenance boundary → W3 quarantine of the level series. (5) Fixings/benchmarks: stamped at the basis prevailing per the administrator's convention at fixing time; consumed at their own basis via as-of replay; transported only along invertible generators, else refused; a fixing spanning a Pending boundary → W1 block until parameter publication (real money never moves on an unwitnessed basis). (6) Curves and all derived data: joint stamp = union of input stamps, inherited COMPOSITIONALLY — the functorial story of Principle re-derivation canon, stated once there and only cross-referenced here; never transported, re-derived in-scope from transported primitives; any input mismatch makes the whole datum inadmissible.

G6 — PLACEMENT + VOICE: one new \subsection{The market-data contract} in §8, immediately after §8.3 (the stamp and the ingest door, tex ~2643), carrying G1's principle + interface, the O1-O7 obligations, the G3 sentence, the G4 table, and the G5 modes + worked example (~3 pp; 168→~171, inside the 200 cap). App D alignment: tex 6655 and Ledger.hs 859/1572 already call mQuoteEx the degenerate per-(unit,source) basis stamp — add the one cross-reference making App D read as the pedagogical one-bit fibre OF the stated contract. Roadmap (tex 89) §8 row gains the boundary-contract pointer; Glossary (tex 6683) gains basis projection, stamp, dissemination convention, market-data contract; FAQ gains "must a provider track corporate actions? No — O2; the stamp says which basis the value is in." First-version voice throughout, no diff-voice; any freshly authored listing swept for agent-name provenance residue (known reintroduction vector). Reference alignment (effTip Ledger.hs 1538, statePrice 871-874) verified by inspection; properties authored in catalogue shape, execution PENDING-TOOLCHAIN.
OBJECTIONS:
1. GATEWAY-STAMPS (Design B) as primary authority: I object unless throughput at the door is PROVEN binding. B converts the stamp from a ledger-computed coordinate into an external assertion; the door must re-verify it anyway under zero-trust, so B is at best a caching optimization and at worst an authority transfer with a standing view-staleness race. If the committee adopts B, it MUST keep full door re-verification, the cited-view-id discipline, and a widened TA-BASIS entry — at which point A with advisory pre-stamp delivers the same throughput relief with a smaller trust surface.
2. PROVIDER-ASSERTED BASIS IN-BAND (self-certified stamps, "the vendor tells us if it's adjusted"): a provider assertion is at most an input to the O3 convention, reconciled daily — never the stamp. mQuoteEx's Boolean was caller-asserted; the whole point of §8 was to stop generalising that trust.
3. STAMP DEFAULTED FROM PREVAILING STATE (stamp = β_t(u) at arrival): already forbidden at tex 2630, and rightly — it erases the lagging-source case and silently reintroduces the phantom. Any design that lets the door "assume current" under load is a security hole, not a degradation mode.
4. EXPORTING OpSpec/ARROWS IN THE PROJECTION so gateways can pre-adjust values: material violation. It invites CA-logic duplication outside the Ledger and derived-data basis laundering (a gateway "helpfully" back-adjusting history is upstream mutable history). The projection must stay arithmetic-free; non-duplication should be enforced by what is exposed, not by instruction.
5. MANDATING A BASIS-INVARIANT VOL PARAMETRIZATION (e.g., "store vols in moneyness so Scale is a no-op"): that is modelling, and a MATERIAL finding if it appears. The framework declares coordinate sensitivity as data and transforms labels only; it must never claim the vol value is invariant, because equivariance of a black box's interior is exactly what the model-free directive forbids relying on.
6. BIDIRECTIONAL ENRICHMENT ("the data layer consults the Ledger for holdings to prioritise subscriptions"): scope creep on the out-arrow. The projection carries basis structure only; position data crossing outward is a different boundary with different confidentiality consequences. Refuse it in this contract; if wanted, it is a separately specified export.
7. TIMESTAMP-JOIN REVIVAL in the modes discussion ("batch rows are old, just use the time"): already refuted at tex 2360-2364; the batch mode MUST be specified as asOf(t_obs) stamping through the same door, or the second ingestion semantics everyone fears gets built by accident.
8. PUTTING THE CONVENTION REGISTRY OUTSIDE THE SPEC as operational config: I object — the stamp's replay determinism cites the convention version, so conventions must be logged events (existing canonical-handler machinery suffices; no new state home, no minimalism cost).

### THORP
DESIGN:
PRACTITIONER VERDICT FIRST. A real market-data team can implement this contract without heroics, on one condition I make normative below: provider-computed values (index levels, vendor curves) stamp as PRIMITIVE observations of the provider's own unit — singleton stamps — and joint stamps are reserved for data the desk derives itself. With that ruling, everything in G1–G5 is transcription of what vendors already publish (adjustment conventions, coordinate metadata, pro-forma files, republication windows), not new intelligence the gateway must invent.

G1 — STAMPING AUTHORITY (named principle). "The Ledger's boundary chain is the single source of basis truth; the ingestion layer stamps by consulting a read-only projection of that chain and never re-implements corporate-action logic." The projection is an interface, a view of the log like any other projection: it exposes (a) the total assignment β as-of a named log position (UnitId -> BasisId, total by Def usBasis/C5); (b) per-unit chain access — boundary ids with t_eff, prec, declaration domains (the effective order of Def basis-id; reference: effTip, Ledger.hs 1538, ledgerBounds 512); (c) per-boundary declared operator kinds, read-only, so market-data ops can run the TA-BASIS daily reconciliation of vendor-published adjustment factors against applied declarations. Consistency semantics, and this is the desk-critical clause: the projection carries a version handle (its as-of log position), and one ingestion run pins one version. Without pinning, a boundary committing mid-file gives a torn stamping run — half a closing file at b3, half at b4, with no record that the split happened. Pinning is cheap (it is how every batch job already reads reference data) and makes the tear unrepresentable. Staleness of the pinned view relative to the world is the already-stated W3 window: detected, not prevented — no new claim.

G2 — PROVIDER OBLIGATIONS (numbered). O1: a raw observation carries value (exact), observation time, source id, and instrument reference in the source's symbology; symbology-to-UnitId mapping is reference-data (out of scope), and an unmappable or unregistered unit is refused at ingest — the existing door already does this. O2: the ingestion layer, not the provider, attaches the basis stamp, from exactly two evidence classes: the source's declared convention under signature (vendors do declare — adjusted/unadjusted field families, adjusted-through dates in file specs, pre-adjustment settings), or gateway assertion against the committed chain via the G1 projection. O3: a datum whose basis cannot be determined from either class is stamped Pending-equivalent and quarantined — never a guessed basis, never defaulted from the ledger's prevailing state (the tex already forbids the default; this is where it bites). O4: providers are NOT required to be corporate-action-aware. A pre-adjusting vendor and a lagging exchange feed are both representable because the stamp says WHICH basis the value is in — mQuoteEx generalised from one bit to the coordinate. This is non-negotiable in practice: US exchanges go ex at the open while overnight venues quote cum through the night; one vendor flips at 00:00 local, another at exchange open; no contract that requires providers to agree on the flip instant is implementable. O5: the per-(unit, source) offset convention ("this source lags the ex transition") is TA-BASIS-owned operational data, maintained by market-data ops and consumed by ingest — it is the codification of what CA ops teams already reconcile daily. O6: the killer case, stated as its own obligation: a HISTORICAL series pulled today arrives back-adjusted — every point is in TODAY's basis regardless of t_obs. The stamp is assigned from the file's declared adjusted-through convention, never from β at t_obs. A gateway that stamps history by observation time mis-bases every pre-boundary point in the file; this is the single most common real-world basis error and the contract must name it.

G3 — DIRECTION OF FLOW, one sentence: "One arrow out — the read-only basis projection, consumable by any ingestion layer; one arrow in — stamped observations through ingest; the market-data layer never writes ledger state, and the Ledger never trusts a self-certified basis beyond the admission checks ingest itself performs." The projection is pull-based and versioned; there is no per-datum interactive query of the Ledger and no callback.

G4 — THE ROWS (market-structure facts; each row: raw datum / stamp coverage / operators acting / fail-closed meaning). (1) SPOT/QUOTES: disseminated trade or quote, minor units, t_obs, source. Stamp: singleton on the quoted unit, per (unit, source). Operators: Scale, Shift, Subst; the lagging-source case is the per-source offset (exchange-open ex mechanics). Fail-closed: unknown convention quarantines the series; W3 partitions by asserted basis around ex-dates. (2) IMPLIED-VOL OBSERVATIONS — strictly observations, no modelling: raw datum is (vol value, expiry, strike coordinate) PLUS transcribed coordinate-system metadata — absolute strike, moneyness, or delta — which vendors publish and the gateway copies, never infers. Stamp: singleton on the underlying unit (option-series units where registered). Operators: under Scale, absolute-strike coordinates relabel by the declared factor — this mirrors what the listed market itself does (option contracts are adjusted for splits and for special distributions above the de-minimis threshold: strikes divided, multipliers adjusted), so the declared arrow is transcription of an exchange fact, not a model; moneyness/delta coordinates carry declared AId under Scale — declared, never assumed, per the OpSpec rule. Under ordinary-dividend Shift boundaries: NO arrow for vol kinds unless the notice declares one, because exchanges do not adjust contracts for ordinary dividends — fail-closed forces fresh post-ex observations, which is exactly what desks do (nobody mechanically transports yesterday's surface across an ex-date). Fail-closed: an unpriced vol point is an enumerated Unpriced, never a model-transported value. Any sticky-strike/sticky-delta content in this row would be vol modelling and is out of contract. (3) INDEX LEVELS: provider-published level is a PRIMITIVE observation of the index unit; singleton stamp {I -> b_I}; the provider is the authority for its own unit, and its rebalance/divisor events enter as basis notices (W4). The joint-stamped divisor of the §8 worked example is the DERIVED case — snapped by us from constituents — and keeps its joint stamp; the two cases are distinguished by who computed the number, and the contract states the criterion once. (4) COMPOSITIONS/WEIGHTS/DIVISORS: provider files; announcement-vs-effective mechanics are real and asymmetric — major providers announce roughly five to nine business days before effectiveness and disseminate pro-forma files ahead of t_eff. The boundary is logged on announcement with future t_eff; pro-forma data stamp at the post-boundary basis id and sit stored-but-inadmissible until the tip advances — no new machinery, C13's pointwise check simply fails until β_t reaches the stamp. Divisor flips applied after the close of t_eff−1 vs at next open across vendors: W3 partitions, by design. Operators: Recompose, invertible iff declared bijective. Fail-closed: the 70.8 phantom of the worked example. (5) FIXINGS/BENCHMARKS: raw datum carries fixing time AND publication time (they differ: overnight-rate fixings publish next morning; the 4pm London FX fix publishes shortly after the window). Stamp: singleton on the benchmark unit. Republication within a correction window (real conventions: same-afternoon and next-morning republication windows exist for the major RFRs) is a NEW observation on the value-quality axis — basis untouched, immutability untouched. Benchmark cessation/succession is representable with existing operators: Subst (unit succession) composed with Shift (the declared fallback spread, a published constant fixed at the cessation announcement) — a named instance, no new OpSpec case. Fail-closed: a fixing for a ceased benchmark with no declared succession arrow meets Terminal — no crossing in either direction. (6) CURVES: derived data, built by us inside a snapshot from transported primitive quotes; the joint stamp is inherited compositionally — the existing functorial story (Re-derivation canon + zipAt), stated once in §8 and REUSED here by reference, not restated. A vendor-supplied curve is the primitive case again: singleton stamp on the vendor-curve unit, same criterion as index levels. Fail-closed: one refusing input refuses the curve point, enumerated per unit.

G5 — OPERATIONAL MODES, one contract three times. Real-time: stamp at ingest from the pinned projection as-of t_obs plus the per-source offset table. End-of-day batch: one pinned projection version per file; the file's declared adjustment convention supplies the stamp evidence; a boundary committing mid-run cannot tear the run (G1 pinning). Historical/back-adjusted pulls: obligation O6 — file-convention stamping, never β_{t_obs}. Multi-vendor: vendors disagreeing on VALUE is data quality, out of scope and ops-owned; vendors disagreeing on BASIS is in scope and already solved — the stamp expresses each vendor's frame and W3 partitions rather than averaging (no median-75-in-no-basis). WORKED EXAMPLE (the two-consumers scenario, to be added alongside the split walk-through): a 2-for-1 split effective at the open; the stored EX1 print 100 is stamped b3. Consumer A snaps at 09:29 — β(u)=b3, the 100 is admissible as-is, marks 1,000×100. Consumer B snaps at 09:31 — β(u)=b4, the SAME stored observation transports along the unique declared arrow to 50, marks 2,000×50. Same series, opposite sides of the boundary, both correct, and the 2,000×100 phantom is a type error in A-side code and a typed refusal at B's seam. STATED CONSEQUENCE, verbatim-grade: "No corporate-action logic exists outside the Ledger, because adjustment arrows are declared once, in the log, and applied by the one evaluator; the ingestion layer's whole competence is transcription — of values, times, and conventions — never adjustment." That sentence is the anti-drift guarantee a desk actually cares about: two gateways, one truth.

G6 — PLACEMENT AND VOICE: a dedicated subsection "The Market-Data Contract" inside §8, immediately after the ingest-door subsection (it completes ingest's story: the door exists at tex 2606; the contract says who stands on the other side). App D needs only its Scope paragraph confirmed — it already reads statePrice/mQuoteEx as the one-step fibre (Ledger.hs Part F says so in-code); add one cross-reference from the fibre to the contract subsection so App D reads as the pedagogical instance OF the stated contract. First-version voice throughout; every row above is written as a fact about market structure, not as a fix.
OBJECTIONS:
Where I would decide differently from the other natural designs, with the trading consequence of each:

(1) AGAINST adjust-on-ingest (normalising all data to the current basis at the gateway). It is the industry's default architecture and it is wrong here: it duplicates CA logic outside the Ledger (two implementations of the split factor WILL drift), it destroys the raw record (making re-stamp repair and the as-known replay impossible), and it breaks on retro-effective boundaries — a late notice would require rewriting already-normalised history, which is exactly the mutation the log forbids. Store raw + stamp, transport at consumption via the one evaluator. The tex already chose this; my addition is that it is also the only VENDOR-COMPATIBLE choice, because vendors will never coordinate their ex conventions and a normalising gateway must therefore encode per-vendor CA interpretations forever.

(2) AGAINST joint constituent stamps on primitive vendor index levels. The purist reading of the Re-derivation canon says an index level is derived data and should carry the full constituent basis map. On a live book that is a 500-wide stamp on every SPX tick and a 500-way pointwise check at every consumption — heroics, and wrong in principle: the provider computed the level inside its own frame, and our log's authority over the provider's arithmetic is nil. Primitive-vs-derived must be "who computed it from OUR stamped inputs", not "is it arithmetically composite". The divisor worked example survives unchanged because that divisor was snapped by us.

(3) AGAINST timestamp-inferred stamping as a fallback ("if no convention, stamp β at t_obs"). Tempting as a coverage-maximiser, and it fails precisely on the highest-volume real case: back-adjusted history, where t_obs is years before the pull and the value is in today's frame. The tex's never-default rule must survive every convenience argument; unknown convention means quarantine, full stop.

(4) AGAINST any per-datum Boolean or enum "adjusted flag" in the G4 rows. The mQuoteEx Bool is pedagogy, and the tex says so; it has no refusal channel and cannot express multi-step chains, leading quotes, or retro-effective inserts. Every row stamps in BasisId or it re-imports the defect the discipline exists to kill.

(5) AGAINST any vol-transport rule beyond coordinate relabelling. A reviewer will be tempted to add "and the surface moves sticky-strike under Scale" or similar. That is a model, it is contested on every desk, and it is out of contract: the framework declares how COORDINATES relabel (an exchange fact) and stays silent on values, forcing fresh observations where no arrow is declared. Any drafted text that predicts a vol value across a boundary is a material finding under the model-agnostic constraint.

(6) AGAINST an interactive per-datum Ledger query from the gateway (the "just call the Ledger" design). Chatty at feed rates, and it creates an unpinnable read: two data in one file stamped against two log positions with no witness. The projection must be pull-based and versioned, one pin per run.

(7) AGAINST putting vendor value-disagreement adjudication in the contract. Cross-vendor tolerance checks, golden-source hierarchies, quorum rules on VALUES are data-quality operations — real, necessary, and out of scope. The contract's only claim is that basis disagreement is resolved by the stamp (W3) BEFORE any value comparison happens; folding quality adjudication in would bloat the boundary and invite the median-across-bases bug back in through ops tooling.
