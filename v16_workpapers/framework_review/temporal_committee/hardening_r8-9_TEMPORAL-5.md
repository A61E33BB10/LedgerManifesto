# Hardening R8-9 — TEMPORAL-5 (LEAD: S5 clock skew vs the three times; S2 deploy-is-orchestration-only)

## S5 — three-times-are-recorded-values-not-wall-clock (LEAD)
The fold orders on (execution, door, hash), ALL recorded values: execution time is read from the payload (arrivals) or record-derived (emissions); door time is assigned by the SINGLE writer's monotonic admission sequence; hash is content-derived. None is a worker clock.

**(i) Two skewed workers admit near-simultaneous events — can skew reorder the fold?** HOLDS. Execution time comes from the payload/record, not the workers' clocks, so skew cannot touch the primary order. Door time is assigned by the ONE door (workers only PROPOSE), so two skewed workers cannot perturb it. Skew can affect only WHICH proposal reaches the single door first → its door-time tiebreak — but that tiebreak orders only COMMUTING same-instant pairs (either order reaches one fold state, harmless) or declared-precedence pairs (deterministic); a non-commuting same-instant pair with no precedence is REFUSED (fails closed, C-2.7), never decided by arrival order. Skew touches neither execution time nor the meaning of the fold.

**(ii) A durable timer fires early/late under skew — identical txid or a fabricated transaction?** HOLDS — identical txid, S4-absorbed. A timer is the liveness half only: the firing's execution time is the RECORD-DERIVED declared date (scheduled instant / record date), NOT the wall-clock fire; monitor time null; the contract reads the RECORDED event + RECORDED state, never the clock. An early/late/duplicate fire keys on the SAME occasion (the same record position) → same causeEventId → same txid → committed once. Skew cannot fabricate a different transaction because no economic content or execution time is clock-derived; it can only delay/duplicate the physical fire.

**(iii) Any BARE wall-clock read entering a recorded time or the fold order?** HOLDS — none. The ONLY clock reads in the system are: monitor time (the Monitor's single read at capture, recorded as provenance in the envelope by the ingestion activity — orders NOTHING, gates NOTHING, null for emissions) and the door's admission stamp (single-writer, monotonic, a tiebreaker only). Execution time — the primary order — is never a clock read. Workflow code uses workflow.Now (deterministic replay clock) for SCHEDULING only, never fed to a contract or a coordinate; a contract cannot even assign execution time (read from payload / record-derived, never edited at the door). A buggy clock-read in workflow code is a determinism violation that degrades to a liveness incident (R-21), never wrong state.

**(iv) Monitor vs execution vs door time under skew — each from a recorded value, never the worker clock?** HOLDS. Execution time: payload/record-derived — skew-immune. Door time: the single door's monotonic stamp (one component, not cross-worker skewed); strict monotonicity is property-tested with the HASH as backstop if it ever fails, so the order stays total and recomputable even under a skewed door clock. Monitor time: the one acknowledged clock read, confined to provenance — a skewed capture clock degrades only the world-vs-ours lateness SPLIT (a diagnostic), never the fold order or txid.

**S5 crux:** the fold order and every transaction identity are functions of recorded values; the only clock reads are monitor-time provenance (orders nothing) and the door's tiebreak stamp (hash-backstopped). Worker skew enters solely at timer FIRING = liveness — a skewed fire yields the identical txid. Skew can delay or duplicate; it can never reorder the fold or fabricate a transaction.

## S2 — deploy-is-orchestration-only
**Confirm a deploy mid-backtest leaves admitted results byte-identical; compute/emit split unaffected by a Build-ID change.** HOLDS. Build-IDs version ORCHESTRATION only, cutover at ContinueAsNew boundaries (old runs drain to their next CAN); economics/model/recipe versions are RECORDED TERMS on the log, independent of the Build-ID (two-axis separation). Already-admitted backtest marks, chain links, and gate decisions are FACTS on the log → read back byte-for-byte, unaffected by any code deploy (S6). The compute/emit split runs the recorded MODEL version (pinned in the derivation request, not Build-ID-derived), so a deploy neither changes which model runs nor re-runs a memoized activity result (in history / on the log). New steps after the cutover run the SAME recorded model version. A deploy changes cadence/sequencing, never mark VALUES or their execution-time order (record-keyed → same txid → byte-identical). The determinism that makes a backtest chain EQUAL production (VM-11) is preserved across the deploy precisely because the coordinates are recorded, not code-derived.

## Verdict
S5 and S2 both HOLD; no BREAK. Both rest on the two settled invariants: the fold and every txid are functions of recorded values on the single hash-chained log (clocks touch only liveness/provenance, never order or identity), and the orchestration/economics-and-model version axes are physically separate (Build-ID orchestration vs recorded model/recipe terms), so neither clock skew nor a deploy can perturb admitted results.
