## S2 — deploy mid-backtest. Claim DEPLOY-IS-ORCHESTRATION-ONLY (a workflow-code deploy touches only Build-ID/axis-1, not the fold; economics on axes 2/3 on the log). **HOLDS.**
(i) **Can a deploy change an admitted (txid, value)? No.** txid = H(input-cut, model-version, recipe/dynamic-version) — none is workflow code, so a deploy that alters HOW/WHEN the model activity is invoked (options, queue, retry policy, sequencing) leaves the txid invariant. The value is the activity's output for those recorded inputs + the recipe version READ FROM THE LOG (not a code constant), so it too is invariant; an already-admitted mark is immutable on the append-only sim log, and a post-deploy recompute re-proposes the same txid → absorbed (first-admitted canonical). **Break iff** a deploy embeds a recipe/model version as a CODE CONSTANT (fusing axis 1 into axis 3) → the value changes; I4 forbids it (the recipe version is read from the log, never compiled in).
(ii) **Can the Build-ID leak into the identity key or the fold? No.** The key derivation is a closed pure function of recorded ledger facts and never reads the Build-ID; the fold is apply∘contract under the recorded ProductTerms version, and a contract fires as a version-pinned activity keyed to THAT RECORDED version — a stale-or-new Build-ID worker still fires the CORRECT recorded economics. **I4 (economics never pinned to Build-ID) IS the block:** the activity's economics version is selected by the recorded lineage, not by which Build-ID orchestrated the call. **Break iff** the contract/model activity version were selected by the workflow's Build-ID rather than routed by the recorded version.
(iii) **In-flight run at deploy — pinned, or mid-history non-determinism? Pinned.** Worker Versioning pins an in-flight run to its Build-ID; a new deploy does not take over in-flight runs — they drain on their Build-ID to the next ContinueAsNew boundary (the standing cutover point), so the backtest run never observes new code mid-history and replays deterministically. Even a MISCONFIGURED deploy that let new code be observed mid-history is caught as a Temporal non-determinism error → the workflow task FAILS (a stuck, visible run), never a corrupted sim ledger (seed R-21: a non-determinism bug degrades to a liveness incident, never wrong ledger state); the stuck backtest re-runs from the sim record, equal by determinism (VM-11). Verdict: I4 + version-pinned economics + Build-ID-pinned runs block every axis-1→axis-2/3 crossing.

## S5 — three-times-are-recorded-values; clock skew across workers cannot reorder the fold. **HOLDS.**
Only recorded EXECUTION time orders the fold (total order = (execution, door, hash)), and execution time is a source-asserted payload value or, for an emitted event, RECORD-DERIVED (the record date / scheduled instant / breaching close's observation time) — NEVER a worker clock. Monitor time is the ONE clock read, but it orders nothing and gates nothing (provenance only); door time is assigned by the SINGLE writer as a strictly-monotonic sequence (not multiple skewed worker clocks), and the event hash keeps the total order even if door-monotonicity glitches. Axis-crossing probe: a Temporal durable timer (axis 1) fires on Temporal's clock, but that firing is NONE of the three times (§sec:substrate) — the emission it produces stamps a RECORD-DERIVED execution time (the scheduled instant read from declared terms on the log), so timer/worker skew shifts only WHEN the emission fires (latency), never the execution time it carries; a late fire yields the identical transaction. Contracts are clock-confined (never fed a wall clock); model activities emit values, not times. **Break iff** an emitted event's execution time were stamped from the timer's wall-clock fire instant instead of the record-derived scheduled instant → skew would perturb the fold order. Containment: clock confinement — the clock is consulted only for non-ordering monitor time; every ordering time is source-asserted or record-derived, and the substrate reads the three times back as data, never writes them.
