# Red-team hardening R4 — TEMPORAL-3 (reviewer, merged into proposal-1)
Per property: verdict, the exact interleaving driven, and the load-bearing precondition that, if violated, flips HOLDS→BREAKS.

## S1 — mid-refold worker crash + model recomputes a different number. Property: REFOLD-ATOMIC. **HOLDS.**
Interleaving driven: a late correction inserts at its execution position; the single writer refolds the tail, appending recomputed transactions + synthesised firings; **crash after a partial tail is appended**; restart re-runs the refold from the insertion point.
Why it holds: the refold is `apply∘contract`, a PURE fold over the ordered log — **it recomputes NO model**. Model outputs are kind-2 re-entered observations read back as LEAVES; a valuation that consumed the corrected leaf is FLAGGED stale (MD-8), never recomputed in-fold — so "a different number on re-run" cannot enter the refold. Crash-resume is idempotent (prop_refoldIdempotent): every appended transaction and synthesised firing carries a cause-derived txid over RECORDED inputs (H(correction,watch,occasion)), never a Temporal run/attempt id, so the re-run absorbs what landed and appends the rest; only the quiescent closure is observable (§sec:substrate). The stale valuation's re-derivation is a SEPARATE forward step, governed by the compute/emit split (model runs once, memoized; door-propose re-presents identical bytes) + first-admitted-canonical.
**Precondition (the break):** the refold must NEVER recompute a model in-fold. If it did, a mid-refold crash + recompute-divergence WOULD leave a pre-crash partial disagreeing with the post-crash re-run — a real break. Doctrine forbids it (MD-8: re-derivation is forward, out-of-fold); enforce, don't assume.

## S7 — namespace failover between pinned cut and door transaction. Property: GATE-STATE-ATOMIC. **HOLDS.**
Sharpest interleaving driven: an active-active global namespace partitions (Temporal split-brain); BOTH regions run the construction workflow, BOTH pin the SAME cut (deterministic from the record), BOTH run the model — getting DIFFERENT numbers — and BOTH propose the (state+decision) transaction.
Why it holds: atomicity and single-writing live at the DOOR (external Transaction Executor), NOT in Temporal's namespace layer. State+decision are ONE door transaction with a cause-derived txid over the pinned cut + recipe version; the door is the single writer PER LINEAGE, credential-bound (R-18/R-20), so both proposals carry the SAME txid → absorbed to exactly-once. No split verdict/state across regions; no second door (failover moves the region, not the door). The inter-region value divergence is the R3 race, contained identically: first-admitted-canonical + reproducibility-class ≤ VM-6. A′=FLAG survives: C is recorded in workflow history, recovered by replay, invariant under failover.
**Versioning-axes non-leak (confirmed):** failover moves the REGION only. Build-ID pins the run's workflow CODE → the failed-over run replays the same history, same cut, same decision; the model/recipe version is a recorded LEDGER lineage fact, identical across regions. Region ≠ Build-ID ≠ recipe version.
**Precondition (the break):** if failover silently changed the Build-ID or let a region run a different recipe default, the two regions would produce STRUCTURALLY different verdicts (not mere numeric drift) under one txid → a GATE-STATE break. Axis separation must be enforced (Build-ID pinned per run; recipe version a lineage fact), so it cannot arise.

**Residual (both).** Correctness rests on two LEDGER-side invariants Temporal must not erode — the door is the single writer per lineage, and the refold/re-derivation split holds. Neither is a Temporal guarantee; both are exactly the preconditions named above.
