# Hardening R6-7 — TEMPORAL-5 (S6 log-is-sole-truth + replay determinism; S3 sandwich carries no workflow state)

## S6 — wipe-and-rebuild determinism

**(i) Is the rebuild a deterministic function of the log alone; can a poisoned/stale cache change rebuilt derived states or gate decisions?** HOLDS. Two kinds of rebuilt object, both immune. (a) Ledger STATE and PROJECTION-kind derived objects (balances, three homes, discount factors, operator-adjusted reads) are a FOLD over the log — `step = apply ∘ contract`, a pure function of the ordered log, reading the log never a live feed (bare-read prohibition). A cache is only a snapshot keyed by the stable (exec,door,hash) triple; a poisoned snapshot is DETECTABLE because it must be bit-identical to the fold from the nearest valid predecessor, and rebuild-from-log (caches wiped) is the ground truth — disagreement is caught, not trusted. (b) RE-ENTERED OBSERVATIONS, GATE DECISIONS, and MARKS are FACTS ON THE LOG, not recomputed — the ledger runs no model on rebuild (C-14.9); they are READ BACK byte-for-byte, so no computation exists for a cache to poison. The only way to change the rebuild is to change the LOG, which the hash chain makes tamper-evident.

**(ii) Does compute/emit-split + env-version-Tier-2 guarantee re-derived outputs match the recorded bytes on rebuild, so a poisoned cache cannot substitute a value?** HOLDS — by a stronger mechanism than the question assumes, and I flag the category slip: **rebuild is Tier-1 READ-BACK, not Tier-2 re-derivation.** On rebuild the model is NEVER re-run; the recorded bytes ARE the value, protected by the hash chain. So a poisoned cache cannot substitute a different value because no re-computation happens that a cache could poison. The env-version + compute/emit split matter only for Tier-2 (an INDEPENDENT party re-deriving to dispute), never for rebuild. This immunity is exactly WHY the architecture forbids the ledger running models: had rebuild re-run the model, a poisoned cache OR a non-deterministic model could substitute — re-entering outputs as observations makes rebuild read-back, immune to both.

**(iii) Does a fabricated txid on rebuild get refused/audit-caught?** HOLDS, two ways. A txid is a pure function of recorded inputs (H(cause,contract,unit,seq) or H(input-cut,model-version,recipe-version)), so a fabricated txid that does not equal H(its claimed inputs) is caught by recomputing H (audit). And a txid injected into the log post-hoc breaks the position-dependent hash chain — full chain re-validation on rebuild detects it. The substrate never writes the log, so it cannot fabricate onto it; a fabricated txid PROPOSED in normal operation is refused at the door (writer discipline/authorisation) or, if structurally valid but economically wrong, caught by recomputation (trust boundary). No fabrication survives rebuild.

**S6 crux:** rebuild = FOLD + READ-BACK of the hash-chained log, never a model re-run; caches are acceleration whose disagreement is detectable; model non-determinism and cache poisoning cannot enter.

## S3 — CAN mid-sandwich / refold

**Confirm: the sandwich carries no workflow state.** HOLDS. The before-mark and after-mark are re-entered observations ON THE LOG; the operator and the certificate are projections; the workflow holds only {unitId, nodeId, log-cursor}. A ContinueAsNew mid-sandwich completes the run and rehydrates from the log: it reads which marks are already on the log and continues FORWARD — reads back a struck mark (never re-strikes; a re-strike recomputes the same txid and is absorbed), strikes an unstruck mark forward. No state is lost because none was carried.

**Double-fire of the post-event mark — window: NONE.** The retroactive sandwich for a LATE CA is struck at its PAST ex-date by the SINGLE WRITER (past-dated synthesis is the writer's work); the substrate re-fires FORWARD only. The two are disjoint in execution time — the writer owns the ex-date strike, the workflow owns forward marks — so they never both strike the ex-date after-mark. Even if they did, the after-mark's txid = H(post-frame cut at ex-date, model-version, recipe-version) is identical for both → the door absorbs the second. No double-fire.

**Half-sandwich — window: NONE observable.** A before-mark without an after-mark is a VALID transient log state, not a broken state: the rehydrated (or refolded) workflow completes it forward. A refold either runs to completion over its tail (only the QUIESCENT closure is observable) or is re-run idempotently from the nearest snapshot — it cannot leave an observable half-sandwich. A permanently-stalled sandwich (lost workflow) surfaces via the overdue-watch sweep (the CA node's out-edge is an armed watch) and via the incomplete/certificate-less chain link (a pending/broken chain, VM-7) — visible, never silent.

**S3 verdict:** HOLDS. No workflow state; retroactive sandwich = writer's past-dated work, forward continuation = substrate's, disjoint in execution time and each idempotent by txid; half-sandwich is transient-valid or visibly overdue, never silent.

## Verdict
S6 and S3 both HOLD; no BREAK. Both reduce to the two settled invariants: the log is the single hash-chained source (rebuild is read-back + pure fold, never a model re-run), and past-dated synthesis is the single writer's work while the substrate re-fires forward-only under txid idempotence.
