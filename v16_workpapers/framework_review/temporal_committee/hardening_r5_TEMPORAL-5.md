## LEAD attack — S4: the door admit as an atomic unique-key insert
The exactly-once claim = the door is a **single linearizable writer** with an **atomic, durable-before-ack, unique-key (txid) append**. It is NOT a property Temporal provides — Temporal only ever delivers at-least-once TO the door; the door collapses it. All three attacks reduce to that invariant.

**(i) Door crash-restarts mid-storm; redelivery after restart.** HOLDS. The append is atomic and durable *before* the ack (write-ahead / quorum-commit), so a crash leaves txid T either fully on the log or absent — never torn; a post-restart redelivery re-reads the log, finds T, absorbs, returns admitted. No double-admit. The only crash-induced failure is a LOST append if a buggy door acked before durability — an under-write, not a double-admit — and even that degrades to latency: the cause event is record-first on the log, so the derivation re-drives, backstopped by the overdue-watch sweep. Named requirement: durable-before-ack.

**(ii) Two workers race the same txid across two DCs (split-brain).** THE CRUX — HOLDS under the invariant, BREAKS only if it is violated. Substrate split-brain is harmless: both Temporal clusters re-drive and propose the same txid to the SAME single door, which serializes and dedupes — exactly why the door sits OUTSIDE Temporal and is the idempotence authority. Door split-brain (two active Transaction Executors committing to one lineage's log) WOULD land two rows — but that directly violates the single-writer discipline the Constitution mandates, caught by the wipe-and-rebuild / single-writer test. Correct multi-region is **active-passive per lineage** (one leader, quorum log: a minority partition cannot commit → no second row) with cross-lineage conflict resolution, never two writers on one lineage. HOLDS under one leader + quorum append; the exactly-once claim IS the single-writer invariant, no weaker.

**(iii) Storm where each retry recomputes a DIFFERENT number.** HOLDS — one row per txid, unconditionally. Under the compute/emit split, proposeToDoor re-presents the ONE memoized payload on every retry → one payload, one row. Even if compute+emit were (wrongly) FUSED so two attempts computed P1≠P2 and both reached the door, the unique-key admits the first and ABSORBS the second at the presence-check before insert → still one row; only the VALUE is then arbitrary (P1 vs P2), the β divergence bounded by the r3 reproducibility class. Two payloads may REACH the door; at most one reaches the INSERT. The check-then-append is atomic by single-writer serialization (no concurrent check-then-act race) or a unique-key constraint. The split's extra guarantee is only that the one row's value is deterministic, not arbitrary-within-β.

## Verdict
All three S4 attacks HOLD; no BREAK under the single-writer invariant. **Crux:** exactly-once is a total function of the log iff the door is a single linearizable writer with an atomic durable-before-ack unique-key append; S1 and S7 both reduce to exactly this. The sole way to break it is two active writers on one lineage — forbidden by the Constitution, caught by the single-writer test. Named requirements for the merged artifact: durable-before-ack; one leader + quorum log per lineage; recipe subsumes seed; env-version out of the identity key (lineage-only).
