# Red-team hardening R5 — S4 retry storm — TEMPORAL-3 (merged reviewer)
Canonical under attack: door admit = ATOMIC UNIQUE-KEY INSERT on the cause-derived txid, key = (input-cut, model-version, recipe/dynamic-version), env-version OUT; exactly-once = total function of the durable log. Per sub-attack: verdict + one-line reason, or exact break case + containment.

## SA1 — storm where every retry computes a DIFFERENT number. **HOLDS (all three ways).**
(a) ONE canonical value: the door insert is atomic unique-key on txid → at most one row per key; a second proposal with the same txid is a no-op. **Two values under one txid is impossible by the door's own atomicity, whatever the payloads.**
(b) One payload per txid reaches the insert — structurally, under the compute/emit split: runModel's result is memoized in history, so every proposeToDoor retry re-presents IDENTICAL bytes; a storm on runModel BEFORE it memoizes never reaches proposeToDoor (which runs only on runModel's recorded completion), so only the single memoized attempt's value flows. Only the FUSED anti-pattern sends >1 payload — and even then (a) admits one. Defense in depth: split (one payload sent) + door insert (one value admitted).
(c) Two txids for one fact is impossible: all three key components are RECORDED facts (input-cut = a pinned log cursor in history; model/recipe/dynamic-version = lineage facts) — none is an attempt-id, wall-clock, or env-version — so every storm retry recomputes the identical txid. **Break iff** a non-recorded input (attempt-id/env-version) leaked into the key → the storm would mint multiple txids per fact → duplicate admissions. Containment: env-version is OUT of the key (canonical); the key is a closed pure function of recorded ledger facts.

## SA2 — two-tier determinism under the storm. **HOLDS.**
Tier-1 read-back is total and storm-invariant: the storm is entirely UPSTREAM of admission; once one value is on the immutable log, read-back is a total function of the log, reproducing the byte unconditionally. The β coverage-check at consumption is unaffected: it reads the admitted observation's attested reproducibility class — a recorded lineage fact fixed AT admission, in the SAME atomic row as the value (one txid) — so it always sees a coherent (value, class) pair; the storm cannot admit attempt-1's value with attempt-2's class.

## SA3 — versioning-axes non-leak into the identity key. **HOLDS.**
A storm that straddles a worker deploy (Build-ID) or an env upgrade does NOT fork the txid: Build-ID/orchestration/env-version are Temporal/infra facts NOT read into the key; the key is closed over recorded ledger facts only. Three axes stay separate — Build-ID (code/replay, NOT identity); model/recipe/dynamic-version (IN the key, these MAKE the fact); env-version (Tier-2 dispute label, OUT of key and code). A storm straddling a genuine MODEL-version change correctly yields two txids (two different facts) — the key doing its job, not a break. **Break iff** the code/env axis leaked into the key → a mid-storm deploy would fork the txid → duplicate admissions.

**Residual (not a correctness break).** The storm's real cost is COMPUTE/LIVENESS: a non-bit-reproducible model may run N times for one admitted value (wasted compute), and unbounded retry could starve the fact. Contained by bounded ScheduleToClose + heartbeat so a live-but-slow model is not storm-retried (T-5 D5), give-up degrading to a visible overdue/blocked item (M-minima), never silence. **Exactly-once ADMISSION is not exactly-once COMPUTATION** — and that is correct, because the admitted number is an observation (what the source said), never a truth.
