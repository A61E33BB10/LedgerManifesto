## S3 — CA valuation sandwich vs a history-limit continue-as-new. Claim SANDWICH-CARRIES-NO-WORKFLOW-STATE. **HOLDS.**
Durable vs derived: the pre/post marks are kind-1 PROJECTIONS (recompute-on-read from the record at their frame-coordinates — nothing durable to half-write or lose); the ONLY durable write is the sandwich CERTIFICATE, a kind-3 recorded event-outcome admitted as ONE door transaction under a cause-derived txid (S4-idempotent). The CA event (the frame-change transaction) + the certificate are the whole durable footprint.
(i) **CAN fires BETWEEN pre-mark and post-mark — no half-sandwich, visible or lost.** The pre-mark is not a durable half — it is a projection; on resume the workflow rehydrates at the CA node (nodeId in the triple, authoritative from the record), RE-COMPUTES both marks as projections at their pinned frame-coordinates, re-assembles, and re-proposes the certificate. Not-yet-admitted → admitted now; already-admitted → same txid → door absorbs (S4). "Certificate half-written" is impossible: the door admits a transaction whole or refuses whole (never a half-applied state). A leg genuinely still owed shows as an un-admitted certificate = a visible overdue obligation (M7 + overdue-watch sweep), re-derived forward — never silently lost.
(ii) **Latent sandwich state a replay could double-apply? None.** The CAN carries ONLY {unitId, nodeId, log-cursor} — no mark values, no partial certificate, no "pre-mark done" flag. Nothing to double-apply; the certificate's cause-derived txid makes any post-CAN re-proposal a no-op. **Break iff** the CAN carried mark/partial-certificate state → a replay could reuse a STALE pre-mark (against a moved market-data cut) → an inconsistent-coordinate certificate. That is precisely the state the claim forbids carrying; enforce triple-only.
(iii) **Post-mark recomputed after CAN and diverges (retry-divergence lens) — sound.** If the post-mark is the pure frame re-coordination (the market data operator projection over the pre-mark), recompute is bit-identical (deterministic over the record) — zero divergence. If MODEL-priced (kind-2), it re-enters under a txid over (input-cut, model/recipe-version): already-admitted → re-proposal absorbed (first-admitted canonical); not-yet → one value admitted (compute/emit split). β at consumption checks the admitted (value, class) pair from the SAME atomic row against ≤ VM-6; a divergence beyond tolerance surfaces as a certificate residual breach (VM-6) → a recorded broken-chain flag (VM-7), never silent. VM-9's ≈0 (three declared tolerances) is thus defended, not asserted.
**Three-times / refold — forward-only.** The CA's ex-date is its as-of (the frame boundary), execution-time-ordered; the sandwich is struck at the CA node in execution order and, on CAN, re-derived FORWARD from the record, never rewound. A LATE CA is a reordering: the single writer refolds, synthesises the retroactive sandwich at the ex-date, and the workflow is signalled to re-read and re-fire forward (never Temporal rewinding). CAN and refold compose — rehydrate the node, re-derive the sandwich forward. The sandwich carries no workflow state because it is a projection pair + one idempotent certificate over recorded facts.

## S6 — LOG-IS-SOLE-TRUTH; poisoned cache smuggling a stale workflow-code version into a rebuilt state. **HOLDS.**
Wipe-rebuild trusts only the log: rebuilt LEDGER state is a fold of the immutable log under contract/recipe versions pinned ON THE LOG (axis 2/3), never the worker's Build-ID (axis 1). A stale/poisoned orchestration-code version can only mis-sequence or delay — a liveness incident degrading to a visible overdue item — never produce wrong state, because a contract fires as a version-pinned activity keyed to the RECORDED ProductTerms version, so a stale-Build-ID worker still fires the CORRECT recorded economics. The substrate stores only schedules-in-flight + orchestration position, never ledger state; a cache that disagreed with the log is caught by prop_refoldEqualsTimely (rebuilt ≠ timely ⇒ ledger state was smuggled into the substrate). **Break iff** economics were pinned to the workflow Build-ID instead of the recorded ProductTerms version (axis 1 fused into axis 2) → a stale worker folds the log under stale economics → wrong rebuilt state. Containment (R-17): economics/recipe versions live on the log and route the version-pinned contract; the orchestration Build-ID never enters the fold. Axis separation IS the anti-poisoning guard.
